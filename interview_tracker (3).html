Maintenance & Internals
MVCC
Rows never overwritten, dead tuples, why VACUUM exists
⚠️ Concept
Instead of overwriting data, Postgres treats every UPDATE as a combination of a DELETE and an INSERT.Because of this, a single logical row in your table might actually have 5 or 10 different physical versions sitting on the disk at the same time.
Because rows are never overwritten, your database accumulates a lot of history. Once a transaction updates or deletes a row, that old version of the row is still sitting on the disk.
Since Postgres doesn't delete data on the fly, it needs a garbage collector. That is exactly what VACUUM is.
INSERT: New live tuple is created.
UPDATE / DELETE : Old tuple is marked dead; new live tuple is created (for updates).
Time Passes: Old transactions finish; tuples become "dead tuples" (bloat).
VACUUM: Runs -> Dead tuples are cleared, opening up space for new inserts.
VACUUM / AUTOVACUUM
Reclaim dead tuples, when to run manually

Table Bloat
What it is, how to detect, VACUUM FULL risk
⚠️ Concept
In PostgreSQL, bloat is the wasted space on your disk occupied by dead tuples that have not yet been cleaned up, or space that was cleaned up but is currently sitting empty inside the table file.
When a table is heavily bloated (e.g., 70% of the file is empty space) and you desperately need that hard drive space back, a standard VACUUM won't help. You need VACUUM FULL.
VACUUM FULL actively packs up the live rows, writes them into a brand-new, perfectly optimized file on disk, and deletes the old bloated file.
WAL
Write-ahead log, crash recovery, replication
⚠️ Concept
The Write-Ahead Log (WAL) is PostgreSQL’s transaction log. It is a continuous, append-only file on disk where Postgres records every single change (inserts, updates, deletes) before those changes are actually applied to the actual table and index files.

INTERNAL COMMANDS BEHAVIOR
SELECT
✅ Deep
When you execute an INSERT statement, Postgres doesn't just append a line to a text file. It treats data in structured 8KB blocks called Pages.
Page Lookup: Postgres checks the Free Space Map (FSM) of the table to find an existing 8KB data page on disk that has enough room for the new row. If none are found, it allocates a brand-new page.
Loading into RAM: The page is loaded from disk into Shared Buffers (Postgres's RAM caching layer) if it isn’t already there.
Writing the WAL: Before touching the data page, Postgres writes an INSERT record to the WAL buffer on disk. This guarantees durability.

UPDATE
✅ Deep
Because Postgres relies on immutable row versions for concurrency, an UPDATE is structurally an atomic combination of a DELETE and an INSERT.
Find and Lock: Postgres locates the existing row page and places a row-level lock on it.
The "Delete" Phase: It writes to the WAL, then updates the old row's xmax header to the current Transaction ID. The old version is now marked for death.
The "Insert" Phase: * It searches the Free Space Map for a page to hold the new version of the row.
The Pointer Chain: Crucially, Postgres links the two rows together. The old row's header points to the physical location (ctid) of the new row. This allows ongoing transactions to traverse the history chain if they need to find the newer or older version of the data.
